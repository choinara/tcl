# /build — 구현 + 테스트 실행

아래 순서를 반드시 따라 실행하라. 순서를 건너뛰지 마라.

## 사용법

```
/build {메뉴코드}          # 컨펌 모드 (기본) — 각 단계 완료 후 사용자 확인
/build {메뉴코드} --auto   # 자율 모드 — 에러/문제 발생 시에만 멈춤
```

예: `/build MM0080`, `/build MM0080 --auto`

## 실행 순서

1. **`scripts/start-task.sh` 실행**
   - 현재 브랜치, 미커밋 변경 확인
   - Flyway 크로스체크: DB 접속 가능 시 `flyway_schema_history` 조회, 접속 불가 시 로컬 기준 + "⚠ DB 미접속 — 로컬 기준" 경고 출력

2. **Plan 문서 확인**
   - `docs/20_peakmate_docs/` 하위에서 해당 메뉴의 `xxx_plan.md` 파일을 찾아 읽기
   - Plan 문서가 없으면 멈추고 "`/plan`을 먼저 실행하세요" 안내

3. **`scripts/new-worktree.sh {메뉴코드}` 실행**
   - 워크트리 생성 (브랜치: `worktree-{메뉴코드}`)

4. **@builder 에이전트 호출** (`isolation: worktree`)
   - **구현 착수 전 필수**: 아래 파일을 반드시 read한 후 구현 시작
     - `docs/claude/백엔드-개발가이드.md` (백엔드 작업 포함 시)
     - `docs/claude/프론트엔드-개발가이드.md` (프론트엔드 작업 포함 시)
     - `docs/claude/파일구조.md` (참조 파일 경로 확인)
     - 유사 기존 메뉴 파일 1개 이상 read (패턴 확인)
   - Plan 문서의 작업 순서대로 구현
   - 각 작업 단계 완료 시 `tsc --noEmit` + `gradlew compileJava`
   - 구현 + 테스트 = 1 작업 단계 (테스트 없이 완료 표시 금지)

   **Edit 도구 사용 시 검증 필수** (#17)
   - 여러 파일에 걸쳐 동일 패턴을 교체할 때는 `replace_all: true` 사용
   - 교체 후 반드시 grep으로 잔존 패턴 확인:
     ```sh
     grep -rn "교체 전 패턴" frontend/src/
     ```
   - grep 결과가 비어 있어야 완료. 남아 있으면 해당 파일을 추가 수정한다

5. **공통 라이브러리 변경 검사** (`/core-change-check all` 실행)
   - peakmate-core, peakmate-core-be 변경 감지
   - 변경이 있는데 사전 보고가 없었다면 **자율 모드라도 멈추고 보고**
   - 사용자 승인 후 진행

6. **Rule 8-1~8-5 자동 검증** (`/rule8-audit all` 실행)
   - AuditableEntity, 에러처리, 시스템로그, 코드중복, 드롭다운 검사
   - FAIL 발견 시 위반 리스트 보고
   - 위반 항목이 Plan에 없는 신규 도입이면 **수정 후 진행**, 기존 코드라면 사용자에게 별도 처리 지시 받음

7. **경계값 불일치 검사** (`/boundary-test` 실행)
   - DB 스키마 ↔ JPA Entity ↔ Request DTO 간 불일치 탐지
   - FAIL 항목(길이 초과 가능, @Valid 미적용 등): 수정 후 진행
   - WARN 항목: 목록 보고 후 사용자 판단

8. **`scripts/complete-task.sh` 실행**
   - 타입체크 + 빌드 + 변경 파일 목록
   - Rule 11 완료 체크리스트(백엔드/프론트엔드별) 모두 통과 확인

9. **UI 캡쳐 + 자체 검열** (`scripts/capture-ui.sh {메뉴코드}` 실행)
   - 6183~6192 순서로 빈 포트를 탐색해 vite dev 서버 시작
   - 캡쳐 완료 후 실행한 포트의 PID를 종료
   - 백엔드(9096)는 실행 중이어야 함. 미실행 시 SKIP하고 이유 보고
   - 로그인 후 해당 메뉴 URL 자동 이동, 3종 스크린샷 저장:
     - `01_full.png` — 전체 페이지
     - `02_viewport.png` — 1440×900 뷰포트
     - `03_mobile.png` — 375×812 모바일
   - 자동 체크리스트 결과 확인:
     - PageTitle 컴포넌트 렌더링 여부
     - 공통 버튼 스타일(mes-btn 계열) 사용 여부
     - AG Grid 헤더 표시 여부
     - 번역키 미노출(한국어 정상 표시) 여부
     - 가로 스크롤 없음 여부
   - **스크린샷 비전 리뷰**: Read 도구로 스크린샷 파일을 읽어 Claude 비전으로 UI/UX 이상 항목 코멘트
     - 검토 기준: 레이아웃 깨짐, 색상 불일치, 버튼·헤더 누락, 텍스트 잘림, 정렬 어긋남, 빈 화면 여부
   - FAIL 또는 비전 리뷰 지적사항 발견 시 → 수정 후 재캡쳐 (수정 없이 커밋 금지)
   - 컨펌 모드: 스크린샷 및 검열 결과 보고 후 "계속하시겠어요?" 확인

10. **워크트리 브랜치에 커밋**
    - 컨펌 모드: "커밋할까요?" 확인 후 커밋
    - 자율 모드: 자동 커밋 (메시지: `feat({메뉴코드}): Plan 기반 구현 완료`)

11. **GS인증 자체 점검 보고 출력**

12. **완료 문서 작성 확인**
    - "구현 완료. 완료 문서를 작성할까요?" 확인
    - 승인 시: `/Users/hschoi/Documents/Git/CHS/docs/20_peakmate_docs/99_완료/{메뉴코드}_{기능명}_완료_{YYMMDD}.md` 작성
      - 필수 포함: 작업일, 작업 배경, 생성/수정 파일 목록, 동작 검증 결과, GS인증 자체 점검 결과
      - 이미 완료 문서가 존재하면 새 파일 생성 금지 — 기존 문서에 "## 수정 이력" 절 추가
    - 거부 시: 작성하지 않고 다음 단계로 진행

13. **사용자에게 완료 보고**
    - "구현 완료. 워크트리 브랜치: `worktree-{메뉴코드}`"
    - "머지하려면 '머지해', 검증하려면 `/menu-verify {메뉴코드}` 입력"

## 구현 규칙 (@builder 적용 기준)

### 구현 원칙 (5단계)

- 실행계획의 모든 내용을 한 줄씩 꼼꼼하게 확인하며 구현. 누락 절대 불허
- 계획과 실제 구현이 다를 경우 임의 판단 금지 — 차이점 정리 후 사용자 컨펌
- 기존 메뉴의 파일 구조·네이밍·API 규칙·화면 구성·UX 흐름까지 동일 패턴 재현
- 공통 컴포넌트·CSS·공통 로직 100% 준수, 임의 신규 패턴 도입 금지
- 기능 단위(DB → API → UI) 하나씩 완료 후 중간 보고, 확인 후 다음 단계
- 계획 문서에 정의된 작업 순서 임의 변경 금지

### Rule 1-10, 12-17 (핵심 요약)

- **Rule 1**: 완료될 때까지 멈추지 마라. 타입체크·빌드 에러 또는 컨펌 필요 시에만 멈춤
- **Rule 2**: any/unknown 금지. 모든 함수에 파라미터·리턴 타입 명시
- **Rule 3**: 각 단계 끝에 `tsc --noEmit` + `gradlew compileJava`. 에러 0인 상태에서만 다음 단계
- **Rule 4**: import 경로 추측 금지. 실제 파일 위치 확인
- **Rule 5**: 기존 테스트 보호. 내 변경으로 실패 시 구현 먼저 점검
- **Rule 6**: 하드코딩 금지. TODO/FIXME/HACK 미완성 표시 금지
- **Rule 7**: 에러 근본 원인 분석. 3회 반복 시 멈추고 보고
- **Rule 9**: 하나의 파일에 여러 기능 동시 추가 금지
- **Rule 10**: 기존 파일 삭제·이동 시 사전 컨펌
- **Rule 12**: 단계 1개 완료마다 커밋. 커밋/푸시는 명시적 지시 필요
- **Rule 15**: SQL Injection·XSS 방어. API마다 인증·인가 체크. 민감 정보 로그 출력 금지
- **Rule 16**: 목록 조회 API는 페이징 또는 가상 스크롤 적용. N+1 쿼리 방지
- **Rule 17**: npm 통일. yarn/pnpm 금지

### 테스트 규칙 (6단계)

- 구현 + 테스트 = 1 작업 단계. 테스트 없이 완료 표시 금지
- 신규 서비스 메서드: 단위 테스트 (정상 + 에러 + 경계값)
- 신규 API 엔드포인트: 컨트롤러 테스트 (200 + 4xx + 5xx)
- 신규 프론트 컴포넌트: 렌더링 + 기본 인터랙션 테스트
- 기존 테스트 실패 시 구현을 먼저 점검. 테스트 수정이 불가피하면 이유 명시
- `./gradlew test` + `npm run test:run` 결과 보고
- 번역 키 추가 시 ko + en 최소 2개 언어 동시 추가

## 자율 모드 중단 조건

`--auto` 모드에서도 아래 상황에서는 **반드시 멈추고 보고**한다:

- `tsc --noEmit` 에러 발생
- `gradlew compileJava` 실패
- Plan에 없는 파일 수정이 필요한 경우
- 에러 3회 반복
- 설계 판단이 필요한 분기점
- **구현 중 예상치 못한 문제(버그, 누락, 호환성) 발견 시** — 진단 보고만 하고 멈춤. 수정은 사용자 지시 후 진행 (핵심 원칙: 진단과 수정 분리)
- **UI 캡쳐 체크리스트 FAIL 또는 비전 리뷰 이상 발견 시** — 지적 항목 보고 후 멈춤. 수정 승인 후 재캡쳐

## 적용 범위

**이 스킬은 메뉴 단위 구현 전용이다.** 아래 작업은 이 스킬로 처리할 수 없다:

| 작업 유형 | 처리 방법 |
|----------|----------|
| 공통코드 정비 (V40 등) | Plan 문서 기반 수동 구현 — `@builder`를 직접 호출하거나 Bash로 마이그레이션 실행 |
| peakmate-core 변경 | 소스 레벨 직접 수정 가능. 변경 후 전체 타입체크 필수 |
| 바이브코딩 규칙 변경 | `/update-rules` 스킬 사용 |
| DB 마이그레이션만 | Plan 문서 기반 수동 구현 — `@builder`를 직접 호출 |
