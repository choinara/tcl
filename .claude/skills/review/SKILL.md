# /review — 실행계획서 검토

바이브코딩 4단계(검토)를 수행한다.

**이 단계를 생략할지 여부를 항상 사용자에게 확인 받아야 한다.**

## 입력

- `$ARGUMENTS[0]` — 검토할 계획서 파일 경로 (예: `docs/.../xxx_plan.md`)

## 검토 규칙

### 4-1. 메모 반영
- 모든 메모를 반영하고 문서를 업데이트
- **아직 구현하지 마. 구현하라고 오더를 전달하기까지 절대 구현하지 마**

### 4-2. 메모의 의도를 정확히 파악
- 메모가 모호하거나 상충하면 임의 해석 금지 — 되물어라
- 메모 간 충돌 발생 시 두 메모 모두 언급하며 처리 방법 확인 요청
- 부정형 메모는 대안 제시, 최종 선택은 사용자

### 4-3. 메모 반영 결과를 항목별로 명시
- 어떤 메모를 어디에 어떻게 반영했는지 매핑 테이블로 표시
- 반영하지 않은 메모가 있으면 이유 명시 (누락 허용 안 됨)

### 4-4. 문서만 업데이트 — 코드 절대 금지
- 이 단계의 "업데이트"는 계획 문서 수정만
- 파일 생성, 코드 수정, 패키지 설치 등 구현 행위 금지

### 4-5. 문서 업데이트 시 변경 전/후 비교
- 기존 내용 삭제 시 diff 형태로 표시
- 대폭 수정된 섹션은 변경 전 버전 별도 표시
- 사소한 변경도 목록에 포함

### 4-6. 명시적 구현 지시가 있을 때까지 대기
- "이 정도면 될 것 같다", "좋아 보인다" 등 긍정 반응은 구현 허가가 아님
- "구현해", "실행해" 등 명확한 지시가 있을 때만 구현 단계로 진입
- "그럼 구현을 시작할까요?" 같은 확인 질문도 하지 마라

### 4-7. 검토 라운드는 여러 번 반복 가능
- 추가 메모가 오면 같은 프로세스 반복
- 이전 라운드에서 반영한 내용을 되돌리라는 메모 올 수 있음
- 각 라운드마다 문서 전체 일관성 재점검

### 4-8. 메모에 없는 내용 임의 추가 금지
- 자체 판단으로 문서에 내용 추가 금지
- 제안이 있으면 문서 외부 텍스트로만 전달
- 사용자가 수용 의사를 밝힌 제안만 반영

## 수행 절차

1. 계획서 전체 읽기
2. Plan 1 자체 검증 체크리스트 13항목 재검증
3. 기존 코드와 대조 — 패턴 일관성 확인
4. GS인증 기준 확인 (입력값 검증, 권한 체크, 에러 메시지 표준화)
5. 보안 인프라 중복 구현 여부 확인
6. 검토 결과 출력

## 출력 형식

```
[검토 결과]

1. 사실 관계 오류: N건
   - ...
2. 누락 항목: N건
   - ...
3. 일관성 문제: N건
   - ...
4. 확인 필요: N건
   - ...

종합: 승인 / 수정 필요
```

## 완료 후

검토 결과만 출력. 코드 수정이나 구현 제안 금지.
