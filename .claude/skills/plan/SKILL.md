# /plan — 실행계획서 작성

바이브코딩 2단계(Plan1 검증) + 3단계(Plan2 상세 계획)를 수행한다.

## 입력

- `$ARGUMENTS[0]` — 메뉴코드 (예: MM0080)
- 사전 조건: 해당 기능의 `_research.md`가 이미 존재해야 한다

## 수행 절차

### 2단계: Plan 1 — 피해야 할 사항 점검

Research를 바탕으로 계획을 수립하기 전, 아래 4가지 함정을 점검한다.

**2-1. 기존 레이어를 무시하는 함수**
- Controller가 Repository를 건너뛰거나 Domain 규칙을 무시한 즉흥 로직
- "한 파일에서 다 처리" 류의 유혹

**2-2. ORM 관례 무시 마이그레이션**
- 엔티티와 불일치하는 컬럼/제약조건
- DDL을 코드 설계와 분리해서 작성
- ORM이 추적할 수 없는 DB 상태 생성

**2-3. 중복 API 엔드포인트**
- 동일한 의미의 API가 여러 URL로 존재
- 메서드(GET/POST)만 다른 유사 중복

**2-4. 기존 도메인 모델과 개념 불일치**
- 기존 도메인 용어와 1:1로 대응 안 되는 엔티티/필드
- 이미 정의된 의미를 재정의

**2-5. 자체 검증 체크리스트 (하나라도 No면 계획 재작성)**

| # | 검증 질문 |
|---|----------|
| 1 | 기존 레이어/도메인 규칙을 깨지 않는가? |
| 2 | ORM-DB-엔티티가 일관되는가? |
| 3 | API 의미 중복이 없는가? |
| 4 | 기존 용어/개념과 충돌하지 않는가? |
| 5 | 트랜잭션 경계가 명확한가? |
| 6 | 읽기/쓰기 책임이 분리되는가? |
| 7 | 권한/보안이 고려되었는가? |
| 8 | 예외/에러 전략이 정의되었는가? |
| 9 | 미래 분리/확장을 방해하지 않는가? |
| 10 | 테스트 가능성이 열려 있는가? |
| 11 | UI에서 사용자가 설정·변경하는 모든 상태값이 DB 컬럼으로 존재하는가? |
| 12 | BE 엔드포인트 제거/통합 시 FE 호출처 grep 검색이 실행계획에 포함됐는가? |
| 13 | customer_code(고객사)와 end_user_code(End User)를 혼용하지 않는가? |

### 3단계: Plan 2 — 상세 구현 계획 작성

별도 마크다운 파일(`xxx_plan.md`)로 작성. 계획 작성 시 2단계 전체를 다시 읽고 자체 검증.

**작성 원칙:**

- **3-1.** 오프셋 페이징 대신 커서 기반 페이징 지원
- **3-2.** 소스파일을 읽고 실제 코드베이스 기반으로 작성
- **3-3.** 기존 코드의 네이밍 컨벤션·디렉토리 구조·모듈 분리 방식 준수. 유사 기능이 있으면 그 패턴 참고
- **3-4.** 기존 코드를 임의 삭제·리팩토링하지 말 것. 요청 범위 밖 수정 금지
- **3-5.** 에러 핸들링과 엣지 케이스 누락 금지 (null/undefined, 빈 배열, HTTP 상태 코드)
- **3-6.** any 타입 금지. API 응답·파라미터·DB 모델에 명확한 타입 부여
- **3-7.** 환경 변수·시크릿 하드코딩 금지
- **3-8.** 한 번에 하나의 변경만 (기능 단위로 나누어 진행)
- **3-9.** 외부 의존성 추가 전 기존 의존성으로 해결 가능한지 먼저 확인
- **3-10.** 코드 변경 후 빌드·테스트·린트 통과 상태 유지
- **3-11.** OWASP Top 10 방지. 인증/인가 로직 우회 금지
- **3-12.** 추측으로 코드 작성 금지. 실제 소스에서 확인 후 작성
- **3-13.** 코딩 규칙·패턴 변경 시 바이브코딩 4개 문서 세트 업데이트를 실행계획 마지막 단계에 포함
